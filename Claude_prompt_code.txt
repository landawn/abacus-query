Problem: Validation is inconsistent. For the same category of parameter, some methods validate it (null check, range/bounds check, emptiness/blank check, sign/positivity check) while comparable methods skip the check entirely — or throw a different exception type for the same violation.

Canonical validators (static helpers on N / CommonUtil): checkArgNotNull/requireNonNull (→ IAE/NPE), checkArgNotEmpty/checkArgNotBlank (→ IAE), checkIndex/checkElementIndex/checkPositionIndex (→ IOOBE), checkFromToIndex/checkFromIndexSize (→ IOOBE), checkArgNotNegative/checkArgPositive (→ IAE). Primitive collections also use rangeCheck (index ∈ [0,size)) and rangeCheckForAdd (index ∈ [0,size]).

What to do:

1. Group methods by validation contract — cluster methods that should share the same rules, using:
  - Method role/semantics (takes an index; an offset/length pair; a fromIndex/toIndex range; a collection/array; a regex/delimiter; a mapping/predicate function; a count/capacity/scale/radix; etc.)
  - Signature shape and parameter types
  - How each parameter is used inside the body (dereferenced, indexed into, divided by, iterated, passed to a sink, invoked, etc.)
2. Map what is validated where — for every method in a group, record which parameters are checked, the kind and exception type of each check, and which methods perform no check.
3. Identify the inconsistencies — flag where peer methods in a group disagree (one validates, another doesn't; or they throw different exception types), and state which behavior is intended/correct, citing existing convention.
4. Produce a fix plan — per inconsistency, recommend the correct check, the canonical method to use, and where to add it. Group the fixes; separately flag any that change observable behavior (e.g. a method that currently tolerates null/negative and would start throwing).
5. Validation order: within a method, checks should generally run in the same order as the parameters appear in the signature; flag out-of-order or after-the-fact validation.

Methodology / guardrails (important for accuracy at this scale):
- Trace one level of delegation before concluding a check is missing — a public overload often validates inside the private impl or sibling method it calls (e.g. split/copy delegate count-checks to N.split/N.copyOfRange).
- Intentional null-tolerance is the library's deliberate convention, not a bug — many methods treat null/empty as empty and return a default. Flag a null-related divergence only when one method throws while a directly comparable sibling silently tolerates it (and then NPEs deep inside or returns a wrong result).
- Watch the conditional-invocation pattern — a function/predicate/mode arg that's only reached on some inputs (empty source, key present) often goes unchecked, so the same null arg throws on one input and passes silently on another.
- Exception-type drift counts as an inconsistency — checkArgNotNull throws IllegalArgumentException, but some Javadocs promise NullPointerException for the same null-arg category.
- Verify every candidate by reading the actual method body — treat any grep/script/agent output as candidates only; sibling primitive classes (IntList/LongList/…) are strong cross-comparison anchors but are also often already consistent.
